## 4. Data Model

### MongoDB Task Schema (`server/models/Task.js`)

```js
{
  _id: ObjectId,                          // Auto-generated by MongoDB
  title: { type: String, required: true, trim: true },
  description: { type: String, default: "" },
  completed: { type: Boolean, default: false },
  priority: { type: String, enum: ["High", "Medium", "Low"], default: "Medium" },
  category: { type: String, default: "General" },
  aiSuggested: { type: Boolean, default: false },  // true if AI populated priority/category
  createdAt: { type: Date, default: Date.now }
}
```

## 5. API Endpoints

### Base URL: `/api/tasks`

| Method | Endpoint | Description | Request Body | Response |
|---|---|---|---|---|
| GET | `/api/tasks` | Fetch all tasks | — | Array of task objects |
| POST | `/api/tasks` | Create a task | `{ title, description }` | Created task (with AI suggestion) |
| PATCH | `/api/tasks/:id` | Toggle complete | — | Updated task |
| DELETE | `/api/tasks/:id` | Delete a task | — | `{ message: "Deleted" }` |

### POST /api/tasks — Flow

```
1. Receive { title, description } from client
2. Call aiSuggest(description) — non-blocking (don't await before saving)
   → Actually: save task optimistically, then update with AI result
3. Save task to MongoDB
4. Return saved task immediately with priority: "Medium" (default)
5. AI result updates the task via a secondary DB write (fire-and-forget pattern)
   OR: await AI call but with a 3-second timeout fallback
```

> **Decision:** Use the timeout pattern — await the AI call with a 3s timeout. If it resolves in time, save with the suggestion. If it times out, save with defaults. This keeps the API response fast and the frontend non-blocked.

## 6. AI Integration

### Service: `server/services/aiSuggest.js`

**What it does:** Takes a task description, sends it to the Claude API, returns a structured JSON with `priority` and `category`.

**Prompt:**
```
You are a task prioritisation assistant. Given a task description, respond with ONLY a valid JSON object — no explanation, no markdown, no extra text.

Format: { "priority": "High" | "Medium" | "Low", "category": "string (e.g. Work, Personal, Urgent, Learning, Health, Finance)" }

Task: "[description]"
```

**Error handling:** If the API call fails or returns unparseable JSON, return `{ priority: "Medium", category: "General", aiSuggested: false }` — never crash the task creation flow.

**Environment variable:** `ANTHROPIC_API_KEY` in `server/.env`

```markdown
## Architectural Decisions

**Why the AI call lives on the server:**
The AI call happens server-side to keep the API key out of the browser, and to
decouple the frontend from any AI provider changes. The frontend never knows
which AI service is being used — it just receives a task object with priority
and category already populated.

**Why optimistic UI:**
Waiting for both the DB write and the AI response before updating the UI creates
a 1–3 second perceived delay. By adding the task to the list immediately and
updating the AI badge asynchronously, the app feels instant — which is the
standard users expect in 2025.

**Timeout fallback on AI:**
The AI call has a 3-second timeout. If Claude doesn't respond in time, the task
is saved with sensible defaults (Medium priority, General category). The app
never fails because of a third-party service.
```
